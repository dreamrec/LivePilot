# Ableton Live 12.3 API Integration Plan

## Context

LivePilot currently targets Live 12.0.25. Versions 12.1 through 12.3.6 introduce significant LOM/MFL API additions that solve real problems we have today and unlock new capabilities. This document catalogs every API change, maps it to LivePilot impact, and defines an implementation plan.

**Current state:** Live 12.0.25, 294 tools, Remote Script + MCP Server + M4L Bridge architecture.
**Target state:** Live 12.3+ support with backward compatibility for 12.0.x users.

---

## Part 1: Complete API Changelog (12.1 → 12.3.6)

### 12.1 (Major Release)

| API Addition | Type | Details |
|---|---|---|
| Auto Shift device | New Device | Monophonic pitch tracking + correction with formant shifting, vibrato LFO, MPE support. Available in all editions. |
| `LooperDevice.export_to_clip_slot(clip_slot_id)` | New LOM Method | Export Looper content to a session clip slot. Requires non-frozen audio track. |
| Tuning System via MFL API | New LOM Access | `live_set.tuning_system` — load, observe, and adjust microtuning systems. |
| Scale awareness for audio clips | New Feature | Audio clips can carry scale metadata. Forwarded to scale-aware devices on the chain. |
| Split Arrangement clips from Clip View | New Command | `Ctrl+Shift+E` / `Cmd+Shift+E` — split arrangement clips at selection. |
| `Track.arrangement_clips` | New LOM Property | Access list of arrangement clips on a track (already used by LivePilot via our handler). |

### 12.1.10

| API Addition | Type | Details |
|---|---|---|
| **`Track.create_midi_clip(start_time, length)`** | **New LOM Method** | Creates empty MIDI clips directly in Arrangement View. Accepts two float arguments: start time and length in beats. This is a native arrangement clip with full envelope support. |

### 12.2 (Major Release)

| API Addition | Type | Details |
|---|---|---|
| **`DeviceParameter.display_value`** | **New LOM Property** | Access parameter value as shown in the UI ("26.0 Hz", "-3.5 dB") through the LOM. Previously only available through M4L bridge `get_display_values`. |
| **`Clip.start_time` observable** | **New LOM Observable** | Watch for clip start time changes in real-time. Enables live arrangement monitoring. |
| **Take Lanes via MFL API** | **New LOM Access** | Create take lanes, access take lane clips, `create_audio_clip` and `create_midi_clip` on take lanes. Take lane names are accessible. |
| `live.banks` parameter mapping notifications | Max Update | Notify when parameter mappings change. |
| Bounce Track in Place / Bounce to New Track | New Commands | Rendering commands (not yet LOM-exposed but exist as UI commands). |
| Scene Follow Actions: Unlinked/Longest modes | New Feature | Follow action can be triggered at end of longest clip or after specified duration. |

### 12.3 (Major Release)

| API Addition | Type | Details |
|---|---|---|
| **`Track.insert_device(name, index?)`** | **New LOM Method** | Insert native Live devices by name at optional position. Only native devices (not plugins/MFL). Name must match browser display name. |
| **`Chain.insert_device(name, index?)`** | **New LOM Method** | Insert devices into rack chains by name. |
| **`RackDevice.insert_chain(index?)`** | **New LOM Method** | Insert new chains into Instrument/Audio Effect Racks. |
| **`DrumChain.in_note` (get/set/observe)** | **New LOM Property** | Control which MIDI note triggers a drum chain. Default -1 = "All Notes". Essential for programmatic Drum Rack construction. |
| Stem Separation | New Feature | Separate audio into stems (drums, vocals, bass, other) via right-click. Available through MFL API ("Note that this feature is also accessible via the Max for Live API"). |
| Device A/B Comparison | New Feature | Toggle between two parameter states (A/B). Shortcut: P. Includes `Compare: Copy A/B to B/A`. |
| Bounce Group Tracks | New Commands | `Bounce Group to New Track` and `Bounce Group in Place`. Pre-mixer, post-FX. Includes return track processing. |
| Paste Bounced Audio | New Command | Copy clips → paste as bounced audio on another track. |

## Part 2: Impact Analysis for LivePilot

### Problems These APIs Solve

| Current Problem | API Solution | Version Required |
|---|---|---|
| Arrangement clips created programmatically don't support automation envelopes | `Track.create_midi_clip` creates native clips with full envelope support | 12.1.10+ |
| Device loading requires browser search + load_browser_item (slow, state-dependent) | `Track.insert_device` inserts by name directly (fast, stateless) | 12.3+ |
| Human-readable parameter values require M4L bridge (`get_display_values`) | `DeviceParameter.display_value` available natively via LOM | 12.2+ |
| No way to build Drum Racks programmatically | `RackDevice.insert_chain` + `DrumChain.in_note` | 12.3+ |
| Can't monitor arrangement changes in real-time | `Clip.start_time` observable | 12.2+ |
| Take lanes/comping not accessible | Full Take Lanes API | 12.2+ |

### New Capabilities Unlocked

| Capability | APIs Used | What It Enables |
|---|---|---|
| **Native arrangement composition** | `Track.create_midi_clip` + `add_new_notes` | Build arrangements with full automation — no more session-clip-duplication workaround. Filter sweeps, volume rides, send automation all work natively. |
| **Instant device loading** | `Track.insert_device(name)` | Load "Reverb", "Auto Filter", "Compressor" in one call without browser search. 10x faster for common operations. |
| **Drum kit construction** | `insert_chain` + `DrumChain.in_note` + `insert_device` | Build custom Drum Racks: create chains, assign MIDI notes, load samples — all through the API. |
| **Stem separation** | MFL API access to stem separation | Split any audio clip into drums/vocals/bass/other — enables intelligent arrangement analysis on existing audio. |
| **A/B parameter comparison** | Device A/B comparison commands | Preview Studio can use this for real A/B testing of device parameter changes. |
| **Comping workflows** | Take Lanes API | Record multiple takes, access/compare them, build comp tracks. |
| **Real-time arrangement awareness** | `Clip.start_time` observable | SongBrain can detect when the user rearranges clips manually — update section purposes and energy arc in real-time. |

## Part 3: Implementation Plan

### Phase 1: Version Detection + Backward Compatibility Layer

**Goal:** Support 12.3+ features while keeping 12.0.x working.

**Files to create/modify:**
- `remote_script/LivePilot/version_detect.py` — detect Live version at startup
- `remote_script/LivePilot/__init__.py` — log detected version
- `mcp_server/runtime/live_version.py` — version capabilities registry
- `mcp_server/runtime/capability_probe.py` — add Live version to probe

**Design:**

```python
# remote_script/LivePilot/version_detect.py
def get_live_version(app):
    """Return (major, minor, patch) from the running Live instance."""
    try:
        version_str = app.get_major_version()  # Returns int like 12
        # Full version from application
        return (12, 3, 6)  # parsed from app properties
    except:
        return (12, 0, 0)  # fallback assumption

def has_feature(version, feature):
    """Check if a feature is available in this Live version."""
    FEATURES = {
        "create_midi_clip_arrangement": (12, 1, 10),
        "display_value": (12, 2, 0),
        "clip_start_time_observable": (12, 2, 0),
        "take_lanes_api": (12, 2, 0),
        "insert_device": (12, 3, 0),
        "insert_chain": (12, 3, 0),
        "drum_chain_in_note": (12, 3, 0),
        "stem_separation": (12, 3, 0),
    }
    required = FEATURES.get(feature, (99, 99, 99))
    return version >= required
```

**Exit criteria:** `get_session_info` response includes `live_version` field. `get_capability_state` reports which 12.3 features are available.

### Phase 2: Native Arrangement Clips (12.1.10+)

**Goal:** Replace session-clip-duplication with native `Track.create_midi_clip` for arrangement composition. This fixes the automation envelope limitation.

**Files to create/modify:**
- `remote_script/LivePilot/arrangement.py` — new handler `create_native_arrangement_clip`
- `mcp_server/tools/arrangement.py` — new MCP tool `create_native_arrangement_clip`
- `mcp_server/tools/arrangement.py` — update `set_arrangement_automation` to work with native clips

**Design:**

```python
# Remote Script handler
@register("create_native_arrangement_clip")
def create_native_arrangement_clip(song, params):
    """Create an empty MIDI clip in arrangement using native 12.1.10+ API.

    Unlike create_arrangement_clip (which duplicates session clips),
    this creates a true native clip with full envelope support.
    """
    track_index = int(params["track_index"])
    start_time = float(params["start_time"])
    length = float(params["length"])

    track = song.tracks[track_index]
    # 12.1.10+ API: Track.create_midi_clip(start_time, length)
    clip = track.create_midi_clip(start_time, length)

    return {
        "created": True,
        "start_time": start_time,
        "length": length,
        "has_envelope_support": True,  # native clips support automation
    }
```

**Migration path:**
- If Live >= 12.1.10: use `create_native_arrangement_clip` for new arrangements
- If Live < 12.1.10: fall back to `create_arrangement_clip` (session clip duplication)
- `set_arrangement_automation` should work on native clips without the STATE_ERROR

**Exit criteria:** Arrangement clips created on 12.1.10+ support `set_arrangement_automation` for volume, pan, send, and device parameters. The Villalobos arrangement workflow works end-to-end without the "cannot create automation envelope" error.

### Phase 3: Direct Device Insertion (12.3+)

**Goal:** Replace browser-search-then-load with single-call device insertion.

**Files to create/modify:**
- `remote_script/LivePilot/devices.py` — new handler `insert_device`
- `mcp_server/tools/devices.py` — new MCP tool or update `find_and_load_device`

**Design:**

```python
# Remote Script handler
@register("insert_device")
def insert_device(song, params):
    """Insert a native device by name at a position (12.3+ API)."""
    track_index = int(params["track_index"])
    device_name = str(params["device_name"])  # e.g., "Reverb", "Auto Filter"
    position = int(params.get("position", -1))  # -1 = end of chain

    track = song.tracks[track_index]
    if position >= 0:
        device = track.insert_device(device_name, position)
    else:
        device = track.insert_device(device_name)

    return {
        "inserted": True,
        "device_name": device_name,
        "position": position,
    }
```

**Migration path:**
- If Live >= 12.3: `find_and_load_device` internally uses `insert_device` (no browser search)
- If Live < 12.3: fall back to current browser search + `load_browser_item` pattern
- Only native devices supported by `insert_device`; plugins still require browser path

**Device name mapping:** Create a registry of canonical device names that `insert_device` accepts:
```
"Reverb", "Delay", "Compressor", "EQ Eight", "Auto Filter",
"Saturator", "Utility", "Limiter", "Gate", "Chorus-Ensemble",
"Phaser-Flanger", "Corpus", "Erosion", "Vinyl Distortion",
"Auto Pan-Tremolo" (renamed in 12.3 from "Auto Pan"),
"Auto Shift" (new in 12.1),
"Drift", "Wavetable", "Analog", "Operator", "Simpler", "Sampler"
```

**Exit criteria:** `find_and_load_device "Reverb"` works instantly on 12.3+ without browser search. Falls back gracefully on 12.0.x.

**Files to create/modify:**

### Phase 5: Drum Rack Construction (12.3+)

**Goal:** Build Drum Racks from scratch — create chains, assign MIDI notes, insert devices.

**Files to create/modify:**
- `remote_script/LivePilot/devices.py` — new handlers: `insert_rack_chain`, `set_drum_chain_note`
- `mcp_server/tools/devices.py` — new MCP tools

**Design:**

```python
@register("insert_rack_chain")
def insert_rack_chain(song, params):
    """Insert a new chain into an Instrument/Audio Effect Rack."""
    track_index = int(params["track_index"])
    device_index = int(params["device_index"])
    position = int(params.get("position", -1))  # -1 = end

    device = song.tracks[track_index].devices[device_index]
    if position >= 0:
        device.insert_chain(position)
    else:
        device.insert_chain()
    return {"inserted": True}

@register("set_drum_chain_note")
def set_drum_chain_note(song, params):
    """Set which MIDI note triggers a drum chain."""
    track_index = int(params["track_index"])
    device_index = int(params["device_index"])
    chain_index = int(params["chain_index"])
    note = int(params["note"])  # MIDI note number, -1 = All Notes

    device = song.tracks[track_index].devices[device_index]
    chain = device.chains[chain_index]
    chain.in_note = note
    return {"chain_index": chain_index, "in_note": note}
```

**Workflow example — build a drum kit:**
1. Load an empty Drum Rack: `insert_device("Drum Rack")`
2. Insert chains: `insert_rack_chain(position=0)` × N
3. Assign notes: `set_drum_chain_note(chain=0, note=36)` (C1 = kick)
4. Load sounds: `insert_device` into each chain (e.g., "Simpler")
5. Load samples: `load_sample_to_simpler` on each chain's Simpler

**Exit criteria:** A complete drum kit can be built from scratch through MCP tools on 12.3+. `/beat` command can optionally use this instead of browser-based kit loading.

## Part 5: Backward Compatibility Strategy

### Principle: Feature Detection, Not Version Gating

Every new API usage wraps in try/except with graceful fallback:

```python
def create_arrangement_clip_smart(song, params):
    """Use native API on 12.1.10+, fall back to session duplication on older."""
    try:
        # 12.1.10+ path: native arrangement clip
        clip = track.create_midi_clip(start_time, length)
        return {"native": True, "has_envelope_support": True}
    except AttributeError:
        # Pre-12.1.10: duplicate session clip
        return create_arrangement_clip(song, params)
```

### Version Reporting

`get_session_info` should include:
```json
{
    "live_version": "12.3.6",
    "api_features": {
        "native_arrangement_clips": true,
        "display_value": true,
        "insert_device": true,
        "take_lanes": true,
        "stem_separation": true
    }
}
```

### Minimum Supported Version

- **12.0.x:** Full core control (293+ tools). No new API features.
- **12.1.10+:** Native arrangement clips with automation. Looper export.
- **12.2+:** Display values natively. Take lanes. Clip observable.
- **12.3+:** Direct device insertion. Drum rack construction. Stem separation.

### Documentation

README and manual should document capability tiers:
- "Core" tier: Live 12.0+
- "Enhanced Arrangement" tier: Live 12.1.10+
- "Full Intelligence" tier: Live 12.3+

## Appendix C: Splice Integration via LivePilot (12.3+)

### How Splice Works in Live 12.3+

Splice is integrated into Ableton's browser as a dedicated **Splice label**. It provides:

1. **Browse & search** — keyword search across Splice's entire catalog
2. **Preview in sync** — samples play back in sync with Live's transport
3. **Search with Sound** — AI-powered: select a clip/region in your session, Splice finds similar sounds based on style and rhythm
4. **Drag & drop** — drag samples from Splice into Session/Arrangement View, Simpler, Sampler, Drum Rack, Drum Sampler, Instrument Rack chains
5. **Auto-license & save** — samples are automatically licensed and saved to `User Library > Samples > Splice`

### What the API Exposes (and Doesn't)

**Available through existing LivePilot tools:**
- Splice samples land in `User Library > Samples > Splice` as regular audio files
- Once downloaded, they're accessible through `search_browser` (searching the Samples category)
- `load_sample_to_simpler` and `load_browser_item` work with Splice-downloaded files
- `read_audio_metadata` works on Splice files (format, duration, sample rate)
- `analyze_loudness` and `analyze_spectrum_offline` work on Splice files

**NOT accessible through the ControlSurface API:**
- Splice's browser UI (it's a web view embedded in Live — not LOM-accessible)
- Splice's search/browse functionality (no API to query Splice catalog from code)
- Search with Sound trigger (requires UI selection + click on the widget)
- Direct drag-and-drop (requires mouse interaction with the Splice panel)
- Splice account/licensing operations

### LivePilot Integration Strategy

Since Splice's catalog search is not API-accessible, our integration focuses on **working with Splice samples after they've been downloaded** and on **guiding the user to use Splice effectively**.

#### Tier 1: Post-Download Sample Intelligence (No New API Needed)

These work today on 12.0+ with downloaded Splice samples:

| Capability | How | Tools Used |
|---|---|---|
| **Find Splice samples in the library** | Search for samples in the Splice folder | `search_browser` with path filter for `Samples/Splice` |
| **Load Splice sample into Simpler** | User downloads sample → we load it | `load_sample_to_simpler` with file path |
| **Analyze Splice sample** | Offline analysis on the downloaded file | `analyze_loudness`, `analyze_spectrum_offline`, `read_audio_metadata` |
| **Compare Splice sample to reference** | Spectral comparison between sample and reference | `compare_to_reference` |
| **Build drum kit from Splice samples** | Load multiple samples into Drum Rack | `insert_rack_chain` + `set_drum_chain_note` + `load_sample_to_simpler` (12.3+ for rack construction) |

**Example workflow:**
```
User: "Find me a warm kick sample from Splice"
Agent: "I'll search your Splice library for kick samples..."
→ search_browser("kick", category="Samples")
→ Filter results to Splice folder
→ analyze_loudness on candidates
→ compare_to_reference against current kick
→ "Found 3 kicks. Sample B has the warmest spectral profile — 
    low-mid emphasis at 120Hz, gentle roll-off. Load it?"
→ load_sample_to_simpler on the chosen one
```

#### Tier 2: Smart Splice Guidance (Skill-Level Integration)

These don't require API access to Splice — they're agent guidance:

| Capability | How | Implementation |
|---|---|---|
| **"Find me a sound like X"** | Guide user to use Search with Sound | Skill guidance: "Select bars 4-8 on the bass track, then click Search with Sound in the Splice panel" |
| **Sample replacement workflow** | Replace a sample intelligently | Analyze current sample → describe target characteristics → guide user to search Splice → load the result |
| **Genre-aware suggestions** | Suggest Splice searches based on SongBrain identity | Use `build_song_brain` identity → suggest search terms: "dark techno hi-hat", "dub techno percussion" |
| **Splice library audit** | Inventory what's been downloaded | Scan `User Library/Samples/Splice/` folder, catalog by type and characteristics |

**Example workflow:**
```
User: "I need a better hi-hat for this track"
Agent: 
  1. analyze current hat pattern (get_notes + analyze_spectrum)
  2. build_song_brain → identity: "dark minimal techno"
  3. "Your current hat is bright (centroid 8.2kHz) and dry. 
     For a darker dub techno feel, try searching Splice for 
     'dark minimal hat closed' or use Search with Sound — 
     select the hat clip and click the Search with Sound widget."
  4. User finds sample in Splice
  5. load_sample_to_simpler → replace old hat
  6. analyze_spectrum → compare before/after
  7. "New hat is warmer (centroid dropped to 4.1kHz). Fits the identity."
```

#### Tier 3: Splice Desktop App Bridge (Future — External Integration)

Splice has a desktop app with its own local API. A future LivePilot integration could:

| Capability | Feasibility | Effort |
|---|---|---|
| Query Splice catalog programmatically via desktop app API | Medium — Splice desktop app has a local SQLite database and REST-like internal API | High — reverse engineering, fragile |
| Auto-download samples matching criteria | Medium — desktop app supports this | High — requires Splice account auth |
| MCP server for Splice | Possible — build a separate Splice MCP server | Separate project, not part of LivePilot core |

**Recommendation:** Tier 3 is out of scope for LivePilot. If Splice ever exposes a public API or MCP server, we integrate. Until then, Tier 1 + Tier 2 is the practical path.

### Implementation Plan for Splice Integration

**Phase 1 (Now — No Live Upgrade Needed):**
1. Add `search_splice_library` helper to `mcp_server/tools/browser.py` — wraps `search_browser` with a path filter for the Splice samples folder
2. Add splice-aware guidance to `livepilot-sounddesign` skill — when the user asks for samples, suggest Splice searches
3. Add splice sample analysis to `livepilot-evaluation` — compare downloaded Splice samples against the current session

**Phase 2 (After 12.3 Upgrade):**
1. Use `insert_device("Simpler")` + `replace_sample` (12.4+) for one-shot Splice sample loading
2. Use `insert_rack_chain` + `set_drum_chain_note` for building drum kits from Splice samples
3. Add `search_with_sound_guidance` to the Wonder Mode — when Wonder suggests a sound change, recommend a Splice Search with Sound selection

**Phase 3 (Future — If Splice Opens API):**
1. Build a `SpliceMCP` server or add Splice tools to LivePilot
2. Catalog search, auto-download, sample preview

### Key Limitation

The core Splice catalog interaction (search, browse, preview, download) happens inside a **web view** embedded in Ableton's browser panel. This web view is not accessible through the ControlSurface API, Max for Live API, or any other programmatic interface. LivePilot cannot search Splice, trigger Search with Sound, or initiate downloads — the user must do these in the Splice panel.

What LivePilot CAN do is everything that happens **before** (guiding what to search for) and **after** (loading, analyzing, comparing, integrating the downloaded sample). This is the honest integration boundary.
