## 2. Architecture — four layers change

### Layer A: `commit_experiment` ledger writer

`mcp_server/experiment/tools.py::commit_experiment` currently executes
the winning branch's compiled plan via `execute_plan_steps_async` and
returns execution results. It does NOT write to `SessionLedger`. v1.21
adds a best-effort ledger write modeled EXACTLY on the v1.20
`apply_semantic_move` pattern (commit `0b3489b`) — same `start_move`
→ `append_action` per success → `finalize_move` shape, same
`kept=(failure_count==0)` + `score=success_fraction` semantics, same
`ledger_entry_id` in the response.

Key differences from `apply_semantic_move`:

- `engine` is determined by the branch's **source**, NOT by escalation
  success. Rule: `engine = "composer" if target.seed and target.seed.source == "composer" else "experiment"`.
  Rationale: a composer-seeded branch that fell back to scaffold (escalation
  returned `ok=False`) is STILL a composer-sourced commit; anti-repetition
  filters that distinguish engines should see it as such. The escalation
  success/fail distinction is already captured in
  `target.evaluation["composer_escalation"]` and doesn't need to double
  up in the engine tag.
- `move_class` = the winning branch's seed family. Where the branch
  has no seed (edge case for scaffold-only commits), use the
  branch's inferred dominant move family from its compiled plan.
- `intent` = `f"{experiment_id}/{branch_id}: {branch.intent or winning_move.intent}"`
  for audit traceability.
- `actions` = one entry per successful step in the executed plan
  (identical shape to apply_semantic_move).

Escalation wrinkle: `commit_experiment` swaps `target.compiled_plan`
between the auditioned scaffold and the full escalated plan before
running the router (see `mcp_server/experiment/tools.py:687-735`). The
ledger write MUST capture actions from the POST-escalation plan (the
actually-executed one via `exec_results`), not the scaffold. Using
`exec_results` (the router's return value) instead of
`target.compiled_plan["steps"]` (which may still point to either plan
depending on control flow) is the correct source of truth. Test
coverage pins both the escalated-success and escalation-fallback
paths explicitly.

### Layer B: SessionLedger reader audit

Both `mcp_server/wonder_mode/tools.py` and
`mcp_server/song_brain/tools.py` read `SessionLedger` via
`ctx.lifespan_context.setdefault("action_ledger", SessionLedger())`.
The question: do either of them ALSO reach for `memory_list` or

The v1.20 director SKILL was pointed at `memory_list` for Phase 1
anti-repetition; that was wrong because `memory_list` reads the
persistent technique library (`memory_learn`-populated), not the
action ledger. If wonder_mode's rescue trigger or song_brain's

just do," they have the same bug.

Scope:
1. Grep all readers for `memory_list`, `get_session_memory`, and
   `action_ledger` access patterns.
2. For each hit, determine intent (recency / persistent-technique /
   session-observation) and assert it's reading the correct store.
3. Fix any mismatches inline (read from the right store).
4. Document the invariant in a comment near each reader:
   `# anti-repetition reads action_ledger; technique library reads memory_list`.
5. Add `tests/test_ledger_readers.py` as a regression guard — scans
   source for readers and checks they import the right store for
   their stated purpose.

# REQUIRED top-level fields:
device_slug: reverb              # str, matches filename stem
device_class_name: Reverb        # str, Ableton's class_name

**Schema enforcement.** `mcp_server/affordances/_schema.py` (a new
module) exports a `validate_preset_file(path: Path) -> list[str]`
function that returns a list of validation error messages (empty
list = valid). The schema test loads each seed YAML and asserts
validation returns `[]`. The loader `resolve_preset()` is tolerant
of missing optional fields but `validate_preset_file` enforces them
at test-time so the corpus stays consistent.

Loader `mcp_server/affordances/presets.py`:

```python
from pathlib import Path
from typing import Optional
import yaml

_AFFORDANCES_ROOT = Path(__file__).parent / "devices"

def resolve_preset(device_slug: str, preset_name: str) -> Optional[dict]:
    """Return the param_overrides dict for a named preset, or None."""
    path = _AFFORDANCES_ROOT / f"{device_slug}.yaml"
    if not path.exists():
        return None
    data = yaml.safe_load(path.read_text(encoding="utf-8"))
    preset = data.get("presets", {}).get(preset_name)
    return preset.get("param_overrides") if preset else None
```

`configure_device` compiler gains `preset` seed_arg handling:

```python
args = kernel.get("seed_args") or {}
preset_name = args.get("preset")
explicit_overrides = args.get("param_overrides", {})

merged = {}
if preset_name:
    slug = args.get("device_slug") or _infer_device_slug(...)
    resolved = resolve_preset(slug, preset_name)
    if resolved is None:
        return _empty_plan(move, [
            f"No preset {preset_name!r} for device slug {slug!r}"
        ])
    merged.update(resolved)
# Explicit overrides win on conflict — user intent beats preset.
merged.update(explicit_overrides)

if not merged:
    return _empty_plan(move, [...])  # existing empty check
```

Back-compat: callers passing only `param_overrides` (no `preset`)
work unchanged. The `preset` arg is additive.

Slug inference: if the caller doesn't pass `device_slug`, the
compiler can't know which preset file to load. v1.21 requires
explicit `device_slug` when `preset` is used; v1.22's full catalog
adds a class_name → slug inference layer.

### Layer D: `configure_record_readiness` move (performance family)

The one new semantic move. Closes the tech_debt entry from v1.20
live test 6 (`set_track_arm(track=2, armed=true)` was raw-tool).

**Contract:**

- `move_id`: `configure_record_readiness`
- `family`: `performance`
- `intent`: `"Arm or disarm a track for recording. When exclusive=True, disarms all other regular tracks first — the standard one-take recording setup."`
- `targets`: `{}` (no creative-dimension claim; performance utility)
- `protect`: `{"signal_integrity": 0.7}` (armed tracks catch stray MIDI + add CPU monitoring cost)
- `risk_level`: `"low"`
- `required_capabilities`: `["session"]` (performance moves require session-mode per existing convention)

**Note on contradictory seed_args (`exclusive=True` + `armed=False`):**
the compiler rejects with an executable=False plan and warning text
`"exclusive=True requires armed=True (the point of exclusive is to
become the single armed track); to disarm individually, call with
exclusive=False"`. The rejection does NOT point at `emergency_simplify`
— that move resets drums+bass balance, not arm state. To disarm all
tracks, call `configure_record_readiness` per track with
`armed=False, exclusive=False`. A future v1.22 `configure_session_arm`
move might bulk-disarm; out of scope here.

**seed_args:**
- `track_index: int` (required)
- `armed: bool` (required)
- `exclusive: bool` (optional, default `False`)

**Compiler steps:**

```
If exclusive=True AND armed=True:
    [CompiledStep(tool="set_exclusive_arm", params={"track_index": track_index}, backend="remote_command")]
Else:
    [CompiledStep(tool="set_track_arm", params={"track_index": track_index, "armed": armed}, backend="remote_command")]
```

Key decision: when `exclusive=True`, use the existing
`set_exclusive_arm` tool which disarms all other tracks AND arms the
target in one call. When `exclusive=False` or `armed=False`, a single
`set_track_arm` call suffices.

Edge case: `exclusive=True` + `armed=False` is a contradiction. The
compiler rejects with warning "exclusive=True requires armed=True (to
become the single armed track); to disarm individually call
configure_record_readiness with exclusive=False". See §3.1 for the
full contract note.

### Why these 4 layers (and no Phase 6 rewrite)

v1.20's Phase 6 decision table is authoritative; adding
`configure_record_readiness` means **one new row** ("Arm a track for
recording" → `configure_record_readiness`) but no structural rewrite.
The escape hatch policy stays word-for-word as shipped.

### 4.1 `commit_experiment` ledger write (commit 2)

**File:** `mcp_server/experiment/tools.py` (modify only)

Add ledger-write block between the existing `commit_branch_async`
call and the response assembly. Shape mirrors
`apply_semantic_move` commit `0b3489b`:

```python
# v1.21: write the committed experiment to the SessionLedger so
# get_last_move / anti-repetition can see it. Best-effort — a
# ledger write failure is logged but does not fail the commit.
ledger_entry_id: Optional[str] = None
try:
    from ..runtime.action_ledger import SessionLedger
    ledger = ctx.lifespan_context.setdefault("action_ledger", SessionLedger())
    # Engine tag reflects branch SOURCE (not escalation success).
    # A composer-sourced branch that fell back to scaffold is still
    # a composer-engine commit.
    engine_tag = (
        "composer"
        if (target.seed is not None and target.seed.source == "composer")
        else "experiment"
    )
    move_class = _infer_move_family(target)  # helper — reads seed.family, falls back to plan inspection
    ledger_entry_id = ledger.start_move(
        engine=engine_tag,
        move_class=move_class,
        intent=f"{experiment_id}/{branch_id}: {target.intent or 'committed winner'}",
        undo_scope="micro",
    )
    # Iterate exec_results (the router's actual execution log) — this
    # captures the POST-escalation plan when escalation fired, because
    # the router was handed the swapped target.compiled_plan["steps"].
    for er in exec_results:
        if er.ok:
            ledger.append_action(
                ledger_entry_id,
                tool_name=er.tool,
                summary=(er.description or er.tool),  # description from the step dict
            )
    ledger.finalize_move(
        ledger_entry_id,
        kept=(failure_count == 0),
        score=(float(success_count) / len(exec_results)) if exec_results else 0.0,
        memory_candidate=False,
    )
except Exception as exc:  # pragma: no cover
    logger.warning("commit_experiment ledger write failed: %s", exc)
```

**Response field addition:** `result["ledger_entry_id"] = ledger_entry_id`
(same pattern as apply_semantic_move).

**Helper `_infer_move_family(target)`:** consults `target.seed.family`
when the seed exists. Fallback: scan `target.compiled_plan["steps"]`
for tool names and map through a small lookup (`set_device_parameter`
→ `sound_design`, `create_clip` → `arrangement`, etc.). Defaults to
"mix" on unknown. Unit-tested in isolation.

   call `get_action_ledger_summary(limit=10)` — 4+ entries. Because
   the consumer tools (`detect_stuckness`, `analyze_composition`)

## 7. Risks + what NOT to do

### Risks

- **Wire-format parity gate fails.** If commit 1 surfaces >3 pre-v1.20
  drift bugs, invoke Approach B': pause v1.21, ship the fixes as
  v1.20.2 patch, restart v1.21 on clean baseline. Do NOT pile drift
  fixes into the v1.21 commit sequence; the patch release keeps the
  history clean.

- **commit_experiment composer-escalation paths.** The
  `compiled_plan` swap between audition and commit means the actions
  recorded in the ledger MUST come from post-escalation execution,
  not from the scaffold. Test coverage must include both paths
  explicitly; mistake here silently under-records what was actually
  committed.

- **Preset merge semantics ambiguity.** The contract: preset
  resolves first, explicit `param_overrides` entries override entry-
  by-entry (last-write-wins at dict-key granularity). Test both
  directions (preset alone; preset + explicit overrides with some
  overlapping and some non-overlapping keys). Don't ship without
  pin tests.

- **exclusive_arm semantics for configure_record_readiness.** The
  existing `set_exclusive_arm` tool disarms all regular tracks.
  Does it affect return tracks (negative indices)? Verify by
  reading `remote_script/LivePilot/tracks.py` for the handler's
  scope. Pin in contract test: the move's `exclusive=True` matches
  `set_exclusive_arm`'s actual scope; don't silently assume.

- **Reader audit false positives.** Some readers may legitimately
  use `memory_list` for surfacing technique history in UI contexts
  (NOT anti-repetition). The audit test must distinguish purpose —
  use comment-based contracts (`# store_purpose: anti_repetition`
  vs `# store_purpose: technique_history`) rather than blanket
  "memory_list is forbidden."

- **Preset YAML missing = runtime error.** If a user calls
  `configure_device({preset: "dub-cathedral"})` and the seed
  library doesn't have `reverb.yaml`, the compiler emits an empty
  plan with a clear warning. Don't crash; don't load partial
  presets. Test the missing-preset path.

### What NOT to do

- **Don't expand the preset library beyond 3 presets.** Scope is
  minimal by design. v1.22 handles the catalog.
- **Don't close the escape hatch.** Observation window hasn't
  elapsed. Doc still says transitional state.
- **Don't refactor `apply_semantic_move`'s ledger write.** The new
  `commit_experiment` code MIRRORS apply's pattern; apply is the
  reference, not the thing being refactored.
- **Don't add new move families.** Same 7 canonical. The performance
  family already exists; `configure_record_readiness` joins it.
- **Don't modify `set_exclusive_arm` or `set_track_arm`.** The
  compiler uses them as-is. Blast radius stays in the new
  compiler + new move.
- **Don't update `phase-6-execution.md` beyond the affordance-
  preset section + a new `configure_record_readiness` contract
  block.** The rest of the doc is correct.
- **Don't add `preset` seed_arg to any other move.** `configure_device`
  is the natural consumer. Other moves either don't need presets
  (`set_track_metadata`) or have their own semantic shape
  (`build_send_chain` seeds device names, not preset names).
