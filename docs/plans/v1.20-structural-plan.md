### Layer 2: Director SKILL.md Phase 6 rewrite

Current Phase 6 mandates "call raw tools + append `add_session_memory`
marker." v1.20's rewrite changes this to:

- **Default**: `apply_semantic_move(move_id, args)` OR
  `create_experiment + commit_experiment(winner)`. The action ledger
  populates automatically — no manual marker required.
- **Escape hatch** (documented as such, NOT preferred): raw tool calls
  permitted ONLY when no semantic_move covers the pattern, AND the
  caller logs BOTH:
  - `add_session_memory(category="move_executed", ...)` — ledger entry
  - `add_session_memory(category="tech_debt", content="no semantic_move
    for <pattern>", ...)` — tracking log for future coverage

## 3. Move-by-move contracts

Full contract specs. The implementation should use the existing
`SemanticMove` dataclass shape + register via `@register` decorator.

### 3.1 Routing family (commit 1)

#### `build_send_chain`
- **family:** `device_creation`
- **targets:** `{"space": 0.4, "depth": 0.3, "cohesion": 0.3}`
- **protect:** `{"low_end": 0.6}` (return chains can lose bass weight)
- **affected_scope:** `{"return_track_index": int, "device_chain": list[str]}`
- **compiler steps:** For each device in `device_chain`:
  1. `load_browser_item` with `track_index = -1 * (return_track_index + 1)`
     (the remote-command convention for returns)
  2. Optional `set_device_parameter` defaults from `affordances/devices/<slug>.yaml`
  3. Final `get_track_info(track_index=-1 * …)` as a verify step (read-only,
     skipped in apply pass)
- **Risk:** MEDIUM — return chains affect all tracks with sends routed.

#### `configure_send_architecture`
- **family:** `mix`
- **targets:** `{"space": 0.5, "depth": 0.3, "cohesion": 0.2}`
- **protect:** `{"clarity": 0.5}`
- **affected_scope:** `{"source_track_indices": list[int], "send_index": int, "levels": list[float]}`
- **compiler steps:** For each `(track_i, level)` pair:
  1. `set_track_send(track_index=track_i, send_index=..., value=...)`
- **Risk:** LOW — per-track send changes are single-parameter and easily undone.

#### `set_track_routing`
- **family:** `mix`
- **targets:** `{}` (routing-internal, no dimension claim)
- **protect:** `{"clarity": 0.5}` (a bad routing move can silence a track)
- **affected_scope:** `{"track_index": int, "output_routing_type": str, "output_routing_channel": str}`
- **compiler steps:** Single `set_track_routing` call
- **Risk:** MEDIUM — routing changes can silence audio or cause feedback.

### 3.2 Device-mutation family (commit 2)

#### `configure_device`
- **family:** `sound_design`
- **targets:** `varies` (inherits from the device's affordance YAML —
  e.g., `darken_reverb` mode claims `{"depth": 0.5, "cohesion": 0.5}`)
- **protect:** `{}` (caller declares via `preset_name`)
- **affected_scope:** `{"track_index": int, "device_index": int, "preset_name": str, "overrides": dict}`
- **compiler steps:** Look up the affordance preset (e.g., `affordances/devices/reverb.yaml`
  preset "dub-cathedral"), then emit a single `batch_set_parameters` call with
  the resolved parameter values.
- **Risk:** LOW-MEDIUM — param values are bounded, but a preset miscoded
  could cancel audio.

#### `remove_device`
- **family:** `sound_design`
- **targets:** `{}`
- **protect:** `{"signal_integrity": 0.9}` (removing a device on a LIVE
  signal path breaks audio; must be guarded)
- **affected_scope:** `{"track_index": int, "device_index": int, "reason": str}`
- **compiler steps:** Single `delete_device(track_index, device_index)` call.
- **Risk:** MEDIUM — destructive; `run_branch_async` must undo cleanly.
- **Note:** Caller's `reason` string goes into session memory for audit.

### 3.3 Content family (commit 3)

#### `load_chord_source`
- **family:** `sound_design`
- **targets:** `{"harmonic": 0.4, "depth": 0.3, "clarity": 0.3}`
- **protect:** `{"cohesion": 0.6}`
- **affected_scope:** `{"track_index": int, "clip_slot": int, "notes": list[dict], "name": str}`
- **compiler steps:**
  1. `create_clip(track_index, clip_slot, length_beats)` — empty MIDI clip
  2. `add_notes(track_index, clip_slot, notes)` — the chord voicing
  3. `set_clip_name(track_index, clip_slot, name)`
- **Risk:** LOW — MIDI clip creation is idempotent and undoable.
- **Typical caller:** Basic Channel dub-chord workflow — load a single
  chord stab as a source for the return chain.

#### `create_drum_rack_pad`
- **family:** `device_creation`
- **targets:** `{"groove": 0.5, "punch": 0.3, "contrast": 0.2}`
- **protect:** `{"cohesion": 0.6}`
- **affected_scope:** `{"track_index": int, "rack_index": int, "pad_note": int, "sample_uri": str}`
- **compiler steps:** Single `add_drum_rack_pad` call with the resolved sample URI.
- **Risk:** LOW — drum rack pads are self-contained within one rack.
- **Typical caller:** Dilla-style drum programming (add kick/snare/hat pads
  one at a time from Chop and Swing pack).

### 3.4 Metadata family (commit 4)

#### `configure_groove`
- **family:** `arrangement`
- **targets:** `{"groove": 0.6, "motion": 0.2, "contrast": 0.2}`
- **protect:** `{"clarity": 0.5}`
- **affected_scope:** `{"track_index": int, "clip_slots": list[int], "groove_template": str, "amount": float}`
- **compiler steps:** For each clip_slot:
  1. `assign_clip_groove(track_index, clip_slot, groove_template)`
  2. `set_groove_params(groove_template, timing_amount=amount)` (once per template)
- **Risk:** LOW-MEDIUM — Dilla-swing level `amount=0.7+` can shift notes
  audibly; caller should verify post-apply.

#### `set_scene_metadata`
- **family:** `arrangement`
- **targets:** `{}` (metadata-only, no dimension claim)
- **protect:** `{}` (no audio impact)
- **affected_scope:** `{"scene_index": int, "name": str | None, "color_index": int | None, "tempo": float | None}`
- **compiler steps:** Conditional based on which fields are provided:
  - `set_scene_name` if name given
  - `set_scene_color` if color_index given
  - `set_scene_tempo` if tempo given
- **Risk:** LOW — no audio impact. Edge case: `set_scene_tempo` DOES
  affect playback timing when the scene is fired; caller must consider.

#### `set_track_metadata` (bundled rename + color)
- **family:** `mix`
- **targets:** `{}` (cosmetic)
- **protect:** `{}`
- **affected_scope:** `{"track_index": int, "name": str | None, "color_index": int | None}`
- **compiler steps:** `set_track_name` + optional `set_track_color`
- **Risk:** LOW — cosmetic, no audio impact.

def test_<move>_compiler_emits_correct_steps():

def test_<move>_handles_invalid_index():

## 7. Risks + what NOT to do

### Risks

- **Register-file conflicts.** 9 new `@register` decorators; a collision
  with existing 33 move_ids would cause silent overwrite in Python
  (dict insertion). **Mitigation:** `test_registry_uniqueness.py` as a
  commit-1 test.
- **Compiler parity.** Each compiler emits plan steps in a specific
  shape (`{"tool": str, "params": dict, "backend": str}`) that
  `execute_plan_steps_async` expects. **Mitigation:** Copy from existing
  `mix_compilers.py` / `device_creation_compilers.py` exactly; don't
  invent a new shape.
- **Escape hatch becoming permanent.** The plan's
  phased-cutover-then-hard-cutover intent can be undermined by a culture
  of "just use the hatch." **Mitigation:** The `tech_debt` log plus a
  commit-5 integration test that scans SKILL.md examples for
  documentation of the escape hatch being used without the required
  markers.
- **Compliance check compatibility.** v1.18.3's `check_brief_compliance`
  maps specific raw tools to dimensions (`_STRUCTURAL_TOOLS`,
  `_RHYTHMIC_TOOLS`, etc.). When the director calls
  `apply_semantic_move`, the compiled plan's per-step tools still match
  these mappings, so compliance fires per step — verify this in live
  test 5.
- **Configure_device preset source-of-truth drift.** The preset name
  (e.g., "dub-cathedral") must resolve to exactly one set of parameter
  values. **Mitigation:** `affordances/devices/<slug>.yaml` is the
  canonical source; compiler reads it at plan-time, not run-time.

### What NOT to do

- **Don't rewrite existing 33 moves to match a new shape.** Add 9 new
  moves using the existing shape. Cleanup of the older moves is separate
  scope (v1.22+).
- **Don't collapse the two memory-log requirements for the escape hatch
  into one.** `move_executed` is a ledger entry (consumed by
  anti-repetition); `tech_debt` is a project-tracking entry (consumed by
  future semantic_move additions). Different consumers.
- **Don't ship without all 6 live pressure tests passing.** Dry-run
  subagent evidence is insufficient — see v1.18.0 retrospective.
- **Don't add new families.** The 7 canonical families
  (`mix`/`arrangement`/`transition`/`sound_design`/`performance`/`device_creation`/`sample`)
  stay. All 9 new moves fit into existing families.
- **Don't modify `execute_plan_steps_async` / `run_branch_async`.** They
  already handle the step shape the new compilers produce. Touching them
  widens the blast radius unnecessarily.
