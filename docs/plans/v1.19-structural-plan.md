## 2. Item A — Experiment state continuity

### Problem

Live-verified in v1.18.0 Test 8: running a 3-branch experiment captures inconsistent `before_snapshot` values because playback state drifts between branches.

```
Branch 1 before_snapshot.track_meters[0].level = 0.764
Branch 2 before_snapshot.track_meters[0].level = 0.000   ← clip stopped?
Branch 3 before_snapshot.track_meters[0].level = 0.873
```

Each branch's before/after comparison is only valid if the `before` state is the same across branches. Right now it isn't.

### Root cause (verified hypothesis)

`mcp_server/experiment/tools.py:run_experiment` runs each branch sequentially: capture before → execute plan → capture after → undo. The `undo()` reverts command history but doesn't guarantee:

- Playback position is the same (if a clip was firing during branch 1's execute, it may have stopped/continued into branch 2's before-capture window)
- No background automation firing
- No scene changes left over from prior branches

### Proposed approach

**Snapshot-and-restore pattern.** Before the first branch runs, capture a "baseline transport state":
- `is_playing`, `song_time`, `playing_clips`
- Track-level `mute`, `solo`, `arm` states
- Any armed clips

Between branches:
1. Call `stop_playback()` to halt transport
2. Restore armed/solo/mute states from baseline
3. Wait a short stabilization window (200-500ms — TBD empirically)
4. Capture `before_snapshot`
5. Run the branch
6. Capture `after_snapshot`
7. `undo()` all successful steps
8. Restore baseline transport state for next branch

### Files to touch

- `mcp_server/experiment/tools.py` — `run_experiment` loop (lines ~260-540)
- NEW helper `mcp_server/experiment/baseline.py` — transport snapshot/restore
- `mcp_server/experiment/engine.py` — possibly add `BranchBaseline` dataclass to `models.py`

### Tests to add

1. `test_experiment_baseline_captured_before_first_branch` — unit test that baseline captures transport state correctly
2. `test_experiment_before_snapshots_are_consistent` — run 3 branches against a deterministic session (loaded test fixture), assert track_meters[0].level stays within 5% between branches' before_snapshots
3. **Live test** — run the v1.18.0 Test 8 scenario (exp with add_space + add_warmth + widen_stereo on the current session). Expected: 3 before_snapshots with consistent track levels.

### Risks

- Ableton may not expose a clean "stop and stabilize" primitive; may need to poll for meter settling
- Adds latency between branches (~500ms each × N branches = 1.5-5s overhead)
- `undo()` in Live 12 may not cover all state changes (e.g., send values set programmatically)
- Edge case: if the user's session has automation running, the baseline may drift even without our changes

### What NOT to do

- **Don't disable undo** — the undo-after-evaluation pattern is what makes experiments safe. Replace `undo()` only with something equivalent.
- **Don't try to freeze automation** — that's a deeper refactor. Accept automation drift as a known limitation.
- **Don't add the baseline snapshot to `BranchSnapshot`** — it's experiment-level, not branch-level. Put it on `ExperimentSet`.

## 3. Item B — Hybrid-packet compilation

### Problem

When user says "Basic Channel meets Dilla swing" or "Villalobos but sparse like Gas", the director currently handles multi-packet references via LLM ad-hoc reasoning — no explicit merge algorithm exists. Observed in v1.18.0 Test 7: the hybrid compilation produced plausible output but depended entirely on LLM improvisation, with no guarantees about contradiction handling.

### Proposed approach

Add `compile_hybrid_brief(packet_ids: list[str], weights: list[float] | None = None)` that produces a merged brief dict. Merge rules:

| Field | Merge strategy |
|---|---|
| `sonic_identity` | UNION (deduplicated) |
| `reach_for.instruments` / `effects` / `packs` | UNION (deduplicated) |
| `avoid` | UNION — both packets' avoids apply |
| `rhythm_idioms` / `harmony_idioms` / `arrangement_idioms` | UNION |
| `evaluation_bias.target_dimensions` | WEIGHTED AVERAGE (default 0.5 each for 2 packets) |
| `evaluation_bias.protect` | MAX per dimension (stricter floor wins) |
| `move_family_bias.favor` | INTERSECTION preferred, UNION fallback when empty |
| `move_family_bias.deprioritize` | INTERSECTION (deprioritize only if BOTH packets do) |
| `dimensions_in_scope` | UNION (widened) |
| `dimensions_deprioritized` | INTERSECTION (only deprioritize if BOTH do) |
| `novelty_budget_default` | MAX — hybrid asks skew exploratory |
| `tempo_hint` | NEAREST-OVERLAP — if ranges overlap, intersect; else midpoint of the gap |

### Handling contradictions

Gas deprioritizes `rhythmic` + Dilla emphasizes `rhythmic` → INTERSECTION gives neither deprioritized, UNION scope includes it. Correct.

Gas tempo 95-115 + Skream-Mala tempo 138-142 → no overlap. Midpoint of gap = 117-120. Report the ambiguity to the user ("Tempo ranges don't overlap — defaulting to midpoint; which anchor do you want?").

### Files to add/touch

- NEW `mcp_server/creative_director/hybrid.py` — the merge function
- NEW MCP tool `compile_hybrid_brief` in `mcp_server/creative_director/tools.py`
- Update `livepilot-creative-director/SKILL.md` Phase 1 to document the new tool
- New reference file `livepilot/skills/livepilot-creative-director/references/hybrid-compilation.md` — document the merge rules + example hybrids

### Tests to add

1. `test_hybrid_compilation_unions_avoid_lists` — merge BC + Dilla, assert avoid has items from both
2. `test_hybrid_compilation_intersects_deprioritized_dimensions` — merge BC (deprioritize rhythmic) + Dilla (deprioritize spatial) → neither deprioritized
3. `test_hybrid_compilation_surfaces_tempo_gap` — merge Gas + Skream-Mala, assert the returned brief notes the tempo-range disjunction
4. `test_hybrid_compilation_3plus_packets` — test that the algorithm works with 3+ packets, not just 2

### Live pressure test

Dispatch a subagent with "make me a track that's Basic Channel meets Dilla" — verify that (a) compile_hybrid_brief gets called, (b) the returned brief has BC's spatial aesthetic + Dilla's rhythmic discipline, (c) no drop-based EDM template creeps in.

### Risks

- 3+ packet hybrids may have weighting ambiguity — default uniform weights; user can override if needed
- `target_dimensions` weighted average may produce a blurred goal vector ("nothing is emphasized")
- Hybrid refs in artist packets' `canonical_genres` may need re-thinking (should artist packets be hybridizable too?)

## 4. Item C — Full #3 architectural fix (director Phase 6 → semantic_move routing)

### Problem (the big one)

v1.18.1 shipped a doc-level fix: director's Phase 6 mandates `add_session_memory(category="move_executed")` after raw-tool execution, so anti-repetition isn't blind. But this is a *discipline* fix — it relies on the director remembering to log. The architectural fix is to **remove the raw-tool-call path entirely** and route all Phase 6 execution through `apply_semantic_move` or `create_experiment + commit_experiment`, so the ledger populates automatically.

### Why this matters

- Anti-repetition becomes reliable (ledger always populated)
- Evaluation becomes comparable (every move gets a semantic_move-level outcome record)
- The director's contract becomes more testable (we can assert that Phase 6 never calls `set_device_parameter` directly — only semantic moves)
- Memory promotion becomes cleaner (verdict + technique → semantic_move pair, no ambiguity)

### Approach — two-phase migration

**Phase 1 (in this work): inventory + add missing semantic_moves.**

Audit the director SKILL.md + Phase 6 docs + the v1.18.0 live-test session logs. Enumerate every tool-call pattern the director uses. Categorize:

| Pattern | Existing semantic_move? | Action |
|---|---|---|
| Load a device on a track | `add_*_device_*` family | Use existing |
| Load a device on a RETURN | None | NEW: `build_send_chain` or similar |
| Set multiple params on a device | None | NEW: `apply_device_preset` or `configure_device` |
| Load a chord clip with notes | None | NEW: `load_chord_source` |
| Configure sends | None | NEW: `configure_send_architecture` |
| Delete a device | None | NEW: `remove_device` (or extend existing) |

Estimate: 5-10 new semantic_moves needed. Each is ~50-80 lines in a new file `mcp_server/semantic_moves/director_moves.py` or similar.

**Phase 2 (in this work): rewrite director Phase 6.**

Replace the raw-tool-call documentation with semantic_move patterns. Remove the v1.18.1 `add_session_memory` mandate — ledger populates automatically now. Update the `livepilot-creative-director/SKILL.md` Phase 6 section.

### Files to touch

- NEW `mcp_server/semantic_moves/director_moves.py` — 5-10 new semantic moves
- `mcp_server/semantic_moves/registry.py` — register the new moves
- `livepilot/skills/livepilot-creative-director/SKILL.md` — Phase 6 rewrite (remove raw-tool path, document semantic_move-based execution)
- `livepilot/skills/livepilot-creative-director/references/anti-repetition-rules.md` — remove the "state-inference fallback" (ledger is now reliable) OR keep it as a fallback for edge cases
- Tests: `tests/test_director_moves.py` (new) and additions to `test_creative_director.py`

### Tests to add

1. `test_director_moves_compile_without_errors` — every new semantic_move compiles against a representative session state
2. `test_director_moves_populate_ledger` — after `apply_semantic_move(director_build_send_chain)`, `get_last_move()` returns the move with correct family + target
3. `test_director_phase_6_does_not_document_raw_tool_calls` — scan director SKILL.md, assert no `set_device_parameter` or `load_browser_item` references outside the "legacy fallback" section
4. **Live pressure test** — re-run the v1.18.0 Test 1 (anti-repetition) scenario. Before fix: ledger was empty because raw calls bypassed it. After fix: ledger populated, anti-repetition works correctly.

### Risks

- **Big surface area** — touching the core creative workflow affects every creative turn
- Edge cases where existing semantic_moves don't cover a Phase 6 pattern — you'll discover these only when live-testing
- Backward compatibility: old sessions / cached plugin versions may expect the raw-tool-call path
- New semantic_moves need their own validation logic (`protected_qualities`, `affected_scope`) — this is non-trivial
- May require `router.classify_step()` updates if any new moves don't fit existing backends

### What NOT to do

- **Don't try to replace the entire `mcp_server/experiment/` pipeline** — experiments already route through semantic_moves via `commit_experiment`. The gap is the direct-execution path.
- **Don't remove `add_session_memory`** — it's still useful for non-move observations (diagnostic notes, user preferences surfaced mid-session).
- **Don't ship without live-testing** — this refactor is risky; pressure-test against at least 3 creative prompts before pushing.

### Decision point for the user

Before starting Item C, ASK THE USER:
- Do they want to keep the legacy raw-tool-call path as an escape hatch (for patterns that genuinely don't fit semantic_moves)?
- Or a hard cutover (director Phase 6 IS semantic_moves, no exceptions)?

Recommend the escape-hatch option for v1.19 — hard cutover can come in v1.20 if semantic_move coverage is proven comprehensive.

## 7. Release discipline per item

For each item shipped, follow the v1.18.x pattern exactly:

1. Work on `vX.Y.Z-dev` branch
2. Commit each item as its own atomic commit (3 items → 3 commits minimum)
3. Merge fast-forward to main
4. Version-bump 13 canonical files. Inline Python loop (copy from the v1.18.3 release commit `8575cac` — search CHANGELOG for the exact file list; the list is also in `CLAUDE.md` under "Version Bump"):
   ```bash
   python3 -c "
   import pathlib
   files = ['package.json','server.json','livepilot/.Codex-plugin/plugin.json',
            'livepilot/.claude-plugin/plugin.json','.claude-plugin/marketplace.json',
            'mcp_server/__init__.py','remote_script/LivePilot/__init__.py',
            'CLAUDE.md','AGENTS.md',
            'livepilot/skills/livepilot-core/references/overview.md',
            'docs/M4L_BRIDGE.md','manifest.json',
            'livepilot/skills/livepilot-evaluation/references/capability-modes.md']
   n = 0
   for f in files:
       p = pathlib.Path(f)
       if not p.exists(): continue
       before = p.read_text()
       after = before.replace('OLD_VER', 'NEW_VER')  # replace both literals
       if after != before:
           p.write_text(after)
           n += after.count('NEW_VER') - before.count('NEW_VER')
   print(f'{n} version strings updated')
   "
   ```
5. `python3 scripts/sync_metadata.py --check` must report "all in sync"
6. CHANGELOG entry — follow the v1.18.3 entry as a template
7. Commit the release-bump + push
8. `gh release create vX.Y.Z --title ... --notes ...`
9. `npm publish`
10. Plugin cache sync: `CACHE=~/.claude/plugins/cache/dreamrec-LivePilot/livepilot; mkdir -p "$CACHE/$NEW_VER" && cp -R livepilot/agents livepilot/commands livepilot/skills "$CACHE/$NEW_VER/" && cp "$CACHE/$OLD_VER/.mcp.json" "$CACHE/$NEW_VER/.mcp.json"`
11. If `remote_script/LivePilot/*.py` changed: run `npx livepilot --install` + call `reload_handlers` MCP tool
12. If `m4l_device/*.amxd` changed: flag to user that Max re-freeze is needed

## 9. What to do if you get stuck

Common stuck signals and the minimum-effective responses:

| Signal | Response |
|---|---|
| Tests pass but live Ableton behaves differently | Run a subagent pressure test. Dry-run logic is insufficient — see v1.18.0 retrospective in CHANGELOG |
| New semantic_move registration doesn't appear in the registry | Check `mcp_server/semantic_moves/registry.py` — new moves need explicit registration via `@register` decorator or file-level inclusion |
| `sync_metadata.py --check` reports drift after you've edited everything | Check `docs/manual/tool-catalog.md` — regenerate with `python3 scripts/generate_tool_catalog.py` |
| You discover a new bug while fixing these items | Document in CHANGELOG Known Issues section of the next release. Don't try to fix everything in one patch. |
| User asks you to bundle multiple items in one release | Push back. v1.18.x pattern was one item per release for a reason — atomic rollback, clear disclosure. |

If genuinely stuck for more than 30 minutes on any single issue, STOP and write a status document. The next session can pick up from there.
