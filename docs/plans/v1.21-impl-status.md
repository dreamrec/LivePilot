## 9. Risk register delta

Beyond the plan's §7 risks, two post-v1.20.1 risks deserve attention:

### 9.1 MCPB manifest drift

The MCPB build script (`scripts/build_mcpb.sh`) post-validates the
embedded `manifest.json` inside the produced bundle. If `manifest.json`'s
`version` doesn't match `${VERSION}` passed to the script, the build
aborts. Mitigation: Step 6.1.1's bump script handles `manifest.json`
first, so by the time Step 6.4.5a runs, the manifest is already at
`1.21.0`. Cross-check with:

```bash
python3 -c "import json; print(json.load(open('manifest.json'))['version'])"
# Expected: 1.21.0 (after bump, before MCPB build)
```

### 9.2 Race condition from v1.20.2 — "Another client is already connected"

Commit `cf019d5` in v1.20.2 added a one-retry layer for the
"Connection closed by Ableton" race condition on the Remote Script's
TCP socket. Live tests 1-4 in plan §5 all dispatch through the MCP
server's single-socket connection. If the retry mechanism has regressed
(unlikely — CI green on v1.20.3), live tests will surface it. Mitigation:
if a live test hangs or reports "Another client already connected", run
`reload_handlers` via MCP (per `feedback_reload_handlers_workflow`
memory) and retry the scenario. If retry still fails, investigate
before continuing — this is NOT a plan-level issue, it's an operational
one.

### 9.3 Analyzer pre-flight wiring in live tests

v1.20.3 made `ensure_analyzer_on_master` a mandatory first step in the
Director Phase 1 Ground sequence. None of v1.21's live-test scenarios
explicitly call this tool because they dispatch through
`apply_semantic_move`, which is executed by the user/director, not by the
plan itself. Mitigation: before running Live Test 1 in Chunk 6.3, the
operator should call `ensure_analyzer_on_master` once. If the analyzer
is missing on master, it auto-loads — so this is zero-cost. The plan
doesn't mention this; add it as a pre-scenario step in Task 6.3.

## Appendix C — Why the baseline drifted (context for future readers)

The v1.21 plan was written on 2026-04-24 in the morning, immediately after
v1.20.1's shipped-same-day CI hardening patch. The plan was committed as
`5434ffb` → `01b781f` (spec-reviewer pass) → `5e207d6` (runbook).

Later the same day, a **five-project live-test campaign** on v1.20.1
surfaced 5 user-facing bugs and 1 race condition (documented at
`~/Desktop/DREAM AI/demo Project/REPORT.md`). These were shipped as
**v1.20.2** (commit `2589fd5`) in 6 atomic bugfix commits with +52 tests.

During the v1.20.2 live-test retrospective, it was noticed that operators
had missed loading `LivePilot_Analyzer` on master in 5 of 5 projects —
a memory-based instruction that humans/LLMs reliably skip. The fix was
an idempotent pre-flight tool (`ensure_analyzer_on_master`), shipped
as **v1.20.3** (commit `045a757`) with +6 tests and a Director SKILL
Phase-1 wiring.

Finally, the v1.20.3 post-release audit (`bece980`) caught that the
README's promised `.mcpb` one-click install had been broken silently
since v1.17 — no build script existed. The sweep created
`scripts/build_mcpb.sh`, shipped the missing `livepilot-1.20.3.mcpb`
bundle to the v1.20.3 release, and then commit `920b58e` mandated the
MCPB build step in every future release checklist.

None of this was foreseeable from the v1.21 plan's authoring context —
the live-test campaign hadn't started yet. The drift is legitimate,
expected, and the mechanical fix-up is small. Future release plans
should consider adding a "Step 0.0 — rebase version literals from
current HEAD" task at worktree creation time to insulate against this
class of drift, or use a helper that reads the current version dynamically
instead of hard-coding source strings in replace pairs.
