## 1. Current Reality

The current implementation direction already has:

- Agent OS phase 1
- a producer-agent prompt
- arrangement tools
- note tools
- theory and harmony tools
- automation tools
- analyzer/perception tools
- memory tools

That means Composition Engine v1 does **not** need to start from zero.

It should be integrated as a structured layer on top of the current stack, not as a fully separate product rewrite.

## 2. What This Integration Should Achieve

The immediate goal is not "complete musical intelligence."

The immediate goal is:

- Claude can take high-level arrangement/composition requests
- Claude can build a useful composition world model
- Claude can run composition-specific critics
- Claude can choose small structural/phrase/gesture moves
- Claude can evaluate whether those moves helped

In practice, the first version should make these requests materially better:

- "turn this loop into a verse"
- "make this section build tension"
- "make this less repetitive"
- "make the automation feel more musical"
- "make the transition feel earned"

## 3. Integration Strategy

Integrate Composition Engine in four slices:

1. `composition state`
2. `composition critics`
3. `composition move planning`
4. `composition evaluation`

Do **not** try to build every advanced concept at once.

The right sequence is:

- get the core state right
- make one or two critics useful
- make a small set of moves reliable
- only then broaden the move library

## 4. What To Build Now

The immediate Composition Engine integration should include exactly these six deliverables.

## 4.1 Composition World Model v1

Build a composition-specific model that extends the current Agent OS world model.

Minimum contents:

- section candidates
- phrase candidates
- track roles
- motif hints
- harmony summary
- automation summary
- tension/contrast proxies

This should be a real internal structure, not just prose generated by Claude.

## 4.2 Composition Critics v1

Implement the first critic set:

- section identity critic
- phrase critic
- repetition critic
- gesture critic
- transition critic

Do not start with every critic from the full spec.

These five are enough to create real value quickly.

## 4.3 Composition Move Library v1

Implement a small high-quality move library first.

Recommended initial move classes:

- subtractive arrangement move
- response phrase move
- cadence improvement move
- transition gesture move
- motif variation move

## 4.4 Composition Evaluator v1

Use the current evaluation philosophy, but specialize it for composition.

The evaluator should score:

- section clarity
- phrase completion
- repetition fatigue reduction
- transition quality
- preservation of identity

## 4.5 Prompt/Runtime Routing

Claude should know when a request is composition-first and route into Composition Engine concepts explicitly.

## 4.6 Outcome Memory For Composition

The first memory upgrade for this work should be composition-outcome storage, not a giant new memory system.

Store:

- composition goal
- context
- move
- evaluation result
- kept/undone

## 5. What To Defer

To keep the integration tractable, defer these:

- full-song planning across many sections
- deep motif graph sophistication
- advanced emotional-arc modeling
- large-scale style transfer
- complex reference-conditioned composition
- multi-agent branching search

Those belong after the first composition loop is reliable.

## 6. Immediate Dependencies

Composition integration depends on a small set of improvements outside the composition code itself.

## 6.1 Required Agent OS Hardening

Before composition integration goes too far, ensure:

- evaluator hard rules are safe
- snapshot normalization is reliable
- measurable/unmeasurable dimensions are honest

This is especially important if composition moves rely on mixed sonic/symbolic evaluation.

## 6.2 Minimal Shared State

Composition Engine does **not** need the full future Project Brain first.

But it does need a minimal shared state object with:

- session summary
- arrangement clips
- track roles
- phrase candidates
- cue points
- automation summaries

This can later become part of Project Brain.

## 9. Build Order

This is the recommended order for Claude implementation.

## Step 1: Models And Contracts

First implement the data models.

Minimum contracts:

- `CompositionState`
- `SectionCandidate`
- `PhraseCandidate`
- `RoleAssignment`
- `CompositionIssue`
- `CompositionMove`
- `CompositionEvaluation`

### Why First

If the data shapes are not explicit, the rest of the implementation will drift into prompt-only behavior.

## Step 2: State Builder

Build the `CompositionStateBuilder`.

Inputs:

- `get_session_info`
- `get_arrangement_clips`
- `get_track_info`
- `get_notes` / arrangement notes
- `analyze_harmony`
- `get_clip_automation`
- `get_onsets`
- `get_novelty`
- `get_momentary_loudness`

Outputs:

- section candidates
- phrase candidates
- role guesses
- harmony summary
- automation summary

### Important Rule

State builder v1 should be conservative.

It is better to produce:

- "unknown section"

than a confidently wrong section label.

## Step 3: Section Inference

Implement section inference next.

### Minimum logic

Use:

- active track count
- bass commitment
- loudness profile
- phrase length
- novelty spikes
- arrangement clip repetition

### Minimum output

- section type guess
- confidence
- evidence list

### First section labels

- loop
- intro-like
- verse-like
- build-like
- drop/chorus-like
- breakdown-like
- unknown

That is enough for v1.

## Step 4: Phrase Inference

Build phrase candidates from:

- bar spans
- note density changes
- harmonic pivots
- novelty peaks
- fill events
- automation turnarounds

### Minimum output

- start bar
- end bar
- cadence strength
- pickup presence
- variation score

## Step 5: Role Inference

Build role inference using:

- track name
- register
- onset density
- rhythmic regularity
- device class
- arrangement context

### Minimum roles

- kick
- bass anchor
- harmony bed
- lead
- rhythmic punctuation
- texture
- transition fx
- utility/unknown

## Step 6: Critics v1

Start with these concrete critic functions:

- `critique_section_identity(state, goal)`
- `critique_phrase_shape(state, goal)`
- `critique_repetition(state, goal)`
- `critique_gestures(state, goal)`
- `critique_transition_strength(state, goal)`

Each should emit:

- issue type
- severity
- confidence
- scope
- evidence
- recommended move classes

## Step 7: Move Library v1

Implement only five moves at first.

### Move 1: Subtractive arrangement move

Examples:

- delay support layer entry
- thin harmony in early bars
- remove competing texture before arrival

### Move 2: Response phrase move

Examples:

- duplicate and vary last motif fragment
- add answer phrase in bars 3-4 or 7-8

### Move 3: Cadence improvement move

Examples:

- improve phrase ending note
- add pickup or turnaround punctuation

### Move 4: Transition gesture move

Examples:

- inhale before arrival
- send throw at phrase end
- narrow then bloom width

### Move 5: Motif variation move

Examples:

- register shift
- rhythmic displacement
- fragment and answer

## Step 8: Planner v1

The planner should rank moves by:

- expected section/phrase improvement
- identity risk
- scope size
- reversibility
- confidence

### Planner rule

Prefer:

- the smallest move that solves the highest-severity issue

## Step 9: Evaluator v1

Build a composition evaluator separate from current mix-style evaluation.

### Minimum scoring dimensions

- section clarity
- phrase satisfaction
- repetition fatigue
- transition strength
- identity preservation

### Important note

Not every one of these will be fully measurable.

That is okay.

The evaluator should still output:

- measured evidence
- inferred evidence
- defer flags when needed

## Step 10: Prompt Integration

Once the state builder, critics, planner, and evaluator exist, update Claude usage.

The producer prompt should:

- invoke Composition Engine for structure-focused requests
- use composition issues to choose moves
- use composition evaluation after structural changes

### Prompt change examples

For:

- "turn this loop into a verse"

Claude should:

1. build composition state
2. inspect section/phrase issues
3. choose one composition move
4. execute
5. evaluate

not jump straight to arbitrary note editing.

## 10.5 Tooling Strategy

You have two viable implementation patterns.

### Option A: Internal Python modules + no new MCP tools yet

Pros:

- faster integration
- lower surface-area growth
- easier iteration

Cons:

- harder to inspect directly from Claude unless the agent wrapper exposes structured state

### Option B: Add explicit composition MCP tools

Examples:

- `build_composition_state`
- `run_composition_critics`
- `evaluate_composition_move`

Pros:

- clearer tool contract
- easier agent orchestration
- easier debugging

Cons:

- larger public tool surface

### Recommendation

For current Claude integration:

- build the logic in internal Python modules first
- expose **at least one** composition-facing tool after the contracts stabilize

The most useful first tool would be:

- `build_composition_state`

## 11. Immediate Acceptance Scenarios

These should be the first benchmark cases.

### Scenario A: Loop to verse

Input:

- an 8-bar loop with too much full-spectrum commitment

Expected behavior:

- infer loop/verse-like role
- detect overexposure
- apply subtractive move
- improve section identity

### Scenario B: Repetition fatigue

Input:

- repeating motif with no variation

Expected behavior:

- detect fatigue
- choose response or motif-variation move
- keep if identity remains strong

### Scenario C: Weak transition

Input:

- section boundary with no pre-arrival preparation

Expected behavior:

- detect weak transition
- add one gesture move
- evaluate improved arrival strength

### Scenario D: Musical automation

Input:

- static support texture with constant tone

Expected behavior:

- detect no phrase-aware gesture
- add one tasteful reveal/conceal/handoff gesture
- avoid over-automation

## 12. Test Plan

## 12.1 Unit Tests

Add tests for:

- section inference
- phrase inference
- role inference
- each critic
- each move ranking rule

## 12.2 Fixture Tests

Create compact fixtures representing:

- sparse loop
- overfull loop
- weak transition section
- repetitive motif section

## 12.3 Integration Tests

Simulate:

1. build composition state
2. run critics
3. select move
4. apply mocked move
5. evaluate result

## 13. What To Avoid During Integration

Do not:

- build every composition critic from the spec immediately
- overfit to one genre
- make giant arrangement edits on move one
- make automation the default answer to every issue
- spend too much time on full-song macro planning before loop/section logic is reliable

## 14. Recommended Delivery Sequence

If Claude is implementing this now, the best sequence is:

1. Phase 0 Agent OS fixes that Composition depends on
2. composition data models
3. composition state builder
4. section inference
5. phrase inference
6. role inference
7. critics v1
8. move library v1
9. composition evaluator v1
10. prompt/runtime integration

That order is much safer than starting with big arrangement mutations.

## 15. Exit Criteria

Composition Engine v1 integration is successful when:

- Claude can route structure-focused requests into composition state/critics
- at least five composition move classes are implemented
- at least three core scenarios regularly produce keepable first or second moves
- composition evaluation exists and is not just free-form prose
- the system improves section identity without rewriting everything

## 16. Summary

For the current implementation moment, the right move is not to build the entire future architecture.

It is to make `Composition Engine v1` real inside the existing stack by:

- adding concrete internal models
- building a conservative composition world model
- implementing a small, strong critic set
- implementing a small, strong move library
- evaluating changes explicitly
- and wiring Claude into that loop cleanly

That is the shortest path from "good spec" to "real compositional intelligence inside LivePilot."
